### 🌟 核心架构思想：大模型驱动的“意图解析与规则引擎”

整体的数据流向如下：
1. **飞书输入** $\rightarrow$ 2. **ESP32-S3 (透传)** $\rightarrow$ 3. **Flask 中控 (LLM 意图解析)** $\rightarrow$ 4. **执行动作 或 设定规则** $\rightarrow$ 5. **反馈给飞书**。

### 一、 场景剖析与实现逻辑

#### 场景 1：直接控制（“帮我打开外卖柜”）
* **用户意图**：即时动作执行 (Immediate Action)。
* **中控处理逻辑**：大模型将这句话解析为 JSON `{"intent": "control", "target": "cabinet_A", "action": "unlock"}`。中控直接向 MQTT 发布开门指令，并返回“已为您打开外卖柜”。

#### 场景 2：查询 + 动态规则设定（“侦测温度，目前多少，若达到100度触发火警”）
* **用户意图**：状态查询 (Query) + 动态规则设定 (Dynamic Rule Setting)。
* **中控处理逻辑**：大模型解析出两件事：
    1.  **查状态**：读取中控缓存的最新温度数据并回复用户。
    2.  **设规则**：解析出规则 `{"intent": "set_rule", "sensor": "temp", "operator": ">=", "value": 100, "action": "trigger_fire_alarm"}`。中控将这个规则存入内存中。当底层 MQTT 传感器不断上报温度时，中控会不断比对这个规则，一旦达标，立刻广播火警。

---

### 二、 中控 (Flask后台) 需要怎么做？

中控是整个系统的核心，需要做两处重大升级：

#### 1. 新增 `/api/agent` 接口 (对接大模型进行意图识别)
中控需要接入一个大模型 API（如智谱清言、通义千问或 OpenAI），用来“听懂”用户的飞书消息。

```python
# 中控端 (Flask) 的核心逻辑框架

active_rules = [] # 存储用户动态设定的规则

@app.route('/api/agent', methods=['POST'])
def agent_endpoint():
    user_text = request.json.get("text")
    current_temp = device_status.get("Temp", "未知") # 假设你存了温度

    system_prompt = f"""
    你是中控AI。当前温度:{current_temp}。
    请解析用户的自然语言，输出严格的JSON格式：
    - 如果是控制，输出: {{"type": "control", "target": "A", "cmd": "UNLOCK"}}
    - 如果是查询，输出: {{"type": "query", "reply": "当前温度是..."}}
    - 如果是设定规则，输出: {{"type": "rule", "condition": ">=", "val": 100, "reply": "已设置监控规则..."}}
    """
    
    # 2. 请求大模型 API (伪代码)
    llm_result = call_llm_api(system_prompt, user_text) 
    
    # 3. 根据 AI 解析的意图执行逻辑
    if llm_result["type"] == "control":
        mqtt_client.publish(PUB_TOPICS[llm_result["target"]], llm_result["cmd"])
        return jsonify({"reply": f"已执行：{llm_result['cmd']}"})
        
    elif llm_result["type"] == "query":
        return jsonify({"reply": llm_result["reply"]})
        
    elif llm_result["type"] == "rule":
        # 把规则存起来
        active_rules.append({"val": llm_result["val"], "condition": llm_result["condition"]})
        return jsonify({"reply": llm_result["reply"]})
```

### 三、 MiniClaw (ESP32-S3) 需要怎么做？

ESP32-S3 在这里的定位是一个 **“透明的数据搬运工 (Transparent Proxy)”**。它不需要做任何复杂的逻辑判断（因为它带不动大模型），它只负责对接飞书和 Flask API。

### 💡 总结与答辩亮点建议

如果你把这个架构做出来了，在写文档或答辩时，一定要强调以下几个专业名词，这会让你的项目逼格瞬间拉满：

1.  **端云协同计算 (Edge-Cloud Collaboration)**：向老师解释，ESP32-S3 算力有限，所以你采用了“边缘侧收集交互（飞书），云端（Flask+大模型）进行语义推理与决策”的架构设计。
2.  **LLM 驱动的规则引擎 (LLM-Driven Rule Engine)**：传统的 IoT 系统规则（比如超过 100 度报警）是写死在代码里的（Hard-coded）。你的系统可以通过**自然语言动态生成运行时规则**，这是真正的 AIoT！
3.  **大语言模型的工具调用 (Function Calling)**：大模型不只是在“聊天”，它在通过输出 JSON 格式，精准地调用中控底层的 MQTT 函数。

按照这个架构去改动，不仅代码量不会太大（主要集中在 Flask 后端的 API 编写），而且逻辑异常清晰，出了 Bug 极易排查。
