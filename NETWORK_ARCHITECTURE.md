# 系统网络通信协议架构 (Network Protocol Architecture)

本系统采用 **“消息总线(MQTT) + 状态决策接口(HTTP)”** 的混合架构，实现了分布式节点的高效协同与逻辑安全控制。



---

## 1. 系统网络拓扑模型

系统核心采用分布式总线架构，各硬件节点与中控后台解耦。

* **消息总线层 (MQTT)**：所有传感器与执行器通过 MQTT Broker (`broker.emqx.io`) 进行异步通信，确保系统高并发下的实时性。
* **逻辑控制层 (HTTP RESTful)**：Web 前端与 AI 智能体通过标准 HTTP 请求与 Flask 后端交互，确保全局上下文查询的原子性与确定性。

---

## 2. 通信协议矩阵

| 链路方向 | 通信协议 | 数据格式 | 作用说明 |
| :--- | :--- | :--- | :--- |
| **设备 → MQTT Broker** | MQTT | UTF-8 文本 | 物理层状态实时上报（心跳、报警、开门） |
| **中控 → 设备** | MQTT | UTF-8 文本 | 远程命令下发（开锁、紧急封锁） |
| **前端 → Flask 中控** | HTTP/AJAX | JSON | 实时状态可视化轮询、管理员权限指令下发 |
| **MiniClaw → Flask 中控** | HTTP/GET | JSON | AI 决策前的上下文上下文获取（查阅安全状态） |

---

## 3. 详细逻辑链路说明

### 3.1 MQTT 总线定义 (Real-time Layer)
各设备遵循约定好的 Topic 规范，保障中控能够统一识别：

* **状态上报 Topic**:
    * `city/status/a` : 外卖柜模块 (设备 A)
    * `city/status/b` : 消防应急模块 (设备 B)
* **指令接收 Topic**:
    * `city/device_A/cmd` : 控制设备 A 执行动作

### 3.2 HTTP 决策接口 (Logic Layer)
为了让 AI (MiniClaw) 具备“判断力”，中控后台通过 HTTP 暴露了如下接口：

* **状态查询**: `GET /api/status`
    * **作用**: AI 在下发控制指令前，必须先调用此接口判断 `emergency` 状态。
    * **响应格式**: `{"A": "正常关闭", "B": "正常", "emergency": false}`
* **操作下发**: `POST /api/send_cmd`
    * **参数**: `{"target": "A", "cmd": "UNLOCK"}`
    * **作用**: 记录操作日志并转发指令至 MQTT。

---

## 4. 安全优先级协议 (Safety Priority Protocol)

本架构设计了一套**基于优先级的中断响应机制**，以应对火灾等极端情况：

1.  **优先级定义**: `火警状态(emergency: True)` 具备最高优先级，系统会自动拒绝一切针对 `A` 设备的开启指令。
2.  **闭环确认**: 
    * 消防模块发布 `火警` 广播。
    * 中控接收并立即更新全局 `emergency` 标志。
    * AI/前端查询到该标志，触发逻辑拦截。
    * 硬件设备收到封锁信号，执行物理互斥保护，确保设备在危险时不可用。

---

## 5. 部署建议
* **MQTT Broker**: 默认使用 `broker.emqx.io:1883`。
* **延迟优化**: Web 前端轮询频率设为 `5000ms`，硬件状态上报周期设为 `2000ms`，确保网络带宽消耗最优。
* **安全性提示**: 在实际部署时，建议为 Topic 添加唯一前缀（如 `group10_`），以防公共 Broker 上的冲突干扰。
