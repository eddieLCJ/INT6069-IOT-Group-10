# L-CASH 系统：跨节点大模型联动逻辑说明文档

## 1. 核心理念：从“自动化”到“认知”
传统的智能家居是**孤岛式自动化**（例如：传感器 A 触发执行器 A）。
**L-CASH** 则是**认知型联动**：LLM 汇总所有 4 个节点的零散数据，将其拼凑成完整的“生活场景图谱”，并基于对用户需求的“理解”下达跨节点的综合指令。

---

## 2. 节点数据结构 (Data Packets)
每个 M5GO 节点每隔固定时间（如 5s）或在状态变化时，通过 MQTT 发送一个 JSON 数据包给后端。

* **Node 1 (身份):** `{ "node": 1, "user": "Grandpa", "event": "entry" }`
* **Node 2 (环境):** `{ "node": 2, "co2": 1850, "temp": 29.5, "light": 150 }`
* **Node 3 (行为):** `{ "node": 3, "tof_dist": 0.15, "gesture": "none" }`
* **Node 4 (状态):** `{ "node": 4, "sos_btn": false, "servo_pos": 0 }`

---

## 3. 大模型决策流 (The Reasoning Loop)

当后端收集到这些数据后，会构建一个**系统提示词 (System Prompt)** 发给大模型：

### 第一阶段：语义聚合 (Context Aggregation)
后端将 JSON 转化为自然语言描述：
> "当前室内用户为爷爷。Node 2 检测到室内二氧化碳极高（1850ppm），环境闷热。Node 3 检测到 ToF 读数从 1.5m 降至 0.15m 且持续平稳。Node 4 目前无手动干预。"

### 第二阶段：逻辑推理 (Reasoning)
LLM 内部进行因果分析：
1.  **高度突降：** 疑似跌倒。
2.  **空气质量差：** 可能是跌倒的诱因（缺氧/晕厥）。
3.  **身份匹配：** 用户是高龄老人，风险等级上调至“紧急”。

### 第三阶段：指令分发 (Action Dispatching)
LLM 生成一套跨节点的执行计划，后端将其拆解为 MQTT 指令发送回各 M5GO：
* **To Node 4:** `SET SERVO_2 90` (打开排风窗)
* **To Node 1:** `PLAY_SOUND "emergency_alert"` (播报警报语音)
* **To Web:** `PUSH_NOTIFICATION "紧急：爷爷疑似在闷热环境中跌倒！"`

## 4. 典型联动场景案例

| 场景名称 | 触发条件 (传感器组合) | LLM 的认知判断 | 联动执行方案 |
| :--- | :--- | :--- | :--- |
| **夜间离场安全** | Node 2 光照极低 + Node 1 RFID 刷卡离家 | 用户在深夜离开家，但室内空气监测仍在运行。 | 降低 Node 2 采集频率以节能；若 Node 4 盆栽缺水，自动浇水。 |
| **无接触健康求助** | Node 3 手势识别为 "Wave" + Node 2 温度偏高 | 用户可能感到不适但不方便起身，通过手势求助。 | Node 4 舵机打开风扇；Node 1 语音确认：“您是否需要联系家人？” |
| **智能防误判控制** | Node 3 ToF 报跌倒 + Node 4 按下“取消”键 | 系统检测到危险，但用户手动消警。 | LLM 记录为“误报”，并调低该时段的报警灵敏度，体现“自适应”特性。 |

---

## 5. 开发建议
1.  **JSON 规范化：** 确保两个负责硬件的同学定义的 JSON 键名一致，方便 AI 同学解析。
2.  **频率控制：** 不要让 4 个节点拼命发数据，只有在数值波动超过阈值时才上报，减轻大模型的 Token 压力。
3.  **延迟容忍：** 考虑到大模型响应需要 1-3 秒，Node 4 的物理 SOS 按钮应设计为“本地直接报警”，不经过大模型，作为安全底层。
